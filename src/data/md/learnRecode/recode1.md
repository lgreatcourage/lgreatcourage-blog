最近在用 Vite + React + TypeScript 搭一个小项目。我越来越确信：React 的关键不在“写页面”，而在“让 UI 可靠地跟随状态变化”。当我不再把注意力放在 DOM 上，而是先问自己“需要哪些状态、由谁持有、如何向下流动”，很多交互会自己长出来。

> 今日关键词：`state` / `props` / `effect` / `ref` / `key` / `CSS Modules`

---

**一个很小但很重要的转变**

以前我喜欢把列表、表单、请求、弹窗、分页都堆进同一个组件里，功能能跑就行。后来发现这会制造“状态洪水”：变量越来越多、依赖越来越乱、改一点会牵一片。现在我更偏向这个拆法：

- 父组件：负责数据请求与状态聚合，决定渲染分支（loading/empty/error）
- 子组件：尽量纯展示，只吃 `props`，把事件通过回调抛回去
- 复杂交互：抽成自定义 hooks，让组件只保留“意图层”的代码

---

**关于 useEffect：把它当作“同步外部世界”**

我以前把 useEffect 当作“生命周期补丁”，渲染完就丢一堆逻辑进去，结果就是依赖数组写不对就无限请求、重复订阅、定时器越开越多。现在我更明确它的职责：

- 当某个依赖变化时，需要去做请求、订阅、计时器、更新标题等“副作用”
- 必须配套清理（unsubscribe / clearTimeout / removeEventListener）
- 复杂依赖先用 `useCallback` 稳住函数引用，可变数据用 `useRef` 承载

我会优先让 effect 更“短”，把细节拆成函数，这样依赖也更清晰。

---

**受控输入：最直观的“状态即 UI”**

我专门练习了一次留言输入框。用 `ref` 存输入值看起来省事，但 UI 不会跟着变，因为 ref 不触发渲染。受控组件的写法反而更稳更可预测：

```tsx
const [message, setMessage] = useState("")

<Search
  value={message}
  onChange={(e) => setMessage(e.target.value)}
  onSearch={(value) => {
    pushDanmaku(value)
    setMessage("")
  }}
/>
```

这样后面要加“字数限制、校验提示、按钮禁用、回车行为统一”，都是顺着 state 往下写。

---

**列表渲染：key 的稳定性是体验细节**

以前我偷懒用 index 当 key，等到列表插入/删除/排序，就会出现难以解释的小问题：焦点跳、动画错位、组件状态被复用错对象。现在我会更谨慎：

- 能用后端 `id` 就用 `id`
- 没有 `id` 就用能稳定标识的字段组合成 key
- 实在不稳定时，宁愿生成一份稳定映射（而不是 index）

---

**派生状态：能算出来就不要再存一份**

另一个让我少踩坑的习惯是：如果某个值可以从已有 state 推导出来，就不要再 setState 保存它。两份数据会变成“同步成本”，迟早出现不一致。能算就算，必要时再用 `useMemo` 做性能优化，但不会为了“看起来专业”而过度缓存。

---

**样式边界：从快到稳**

Tailwind 写得快，但当结构复杂、类名堆叠时，阅读压力会明显上升。我试着把页面样式迁到 CSS Modules 以后，感觉像把组件“关进自己的房间”：

- 样式隔离减少全局冲突
- 重构组件时不容易误伤别处
- 结构与样式职责更清楚，长期更耐改

---

**下一步**

- 把请求层收口：统一 axios 实例、拦截器、错误处理、数据格式化
- 把常用交互沉淀为组件与 hooks：列表加载、空态、表单校验、弹幕/动画等

这轮学习最大的收获不是记住了多少 API，而是我开始能解释：为什么要这样拆组件、为什么依赖数组要这样写、为什么 state 的形状决定了 UI 的稳定性。继续写下去，React 会越来越像一套“工程习惯”，而不是一堆技巧。
